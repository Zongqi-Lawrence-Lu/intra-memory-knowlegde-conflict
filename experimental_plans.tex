\section{Model and Training Pipeline}

\subsection{Architecture and Tokenizer}

\begin{itemize}
  \item Model: GPT-2-small, 12 layers, 768 hidden dim, 12 attention heads,
    $\sim$117M parameters \citep{radford2019language}. GPT-2-medium (24 layers,
    1024 hidden dim, 16 heads, $\sim$345M) is reserved for confirmatory runs on
    conditions selected by the pilot in \S\ref{sec:seeds}.
  \item Tokenizer: pretrained GPT-2 BPE, used unmodified.
  \item Context length: 512 tokens, reduced to 256 if corpus statistics support
    it. Positional embeddings are trained from scratch, so context length is
    not constrained by the pretrained GPT-2 checkpoint.
\end{itemize}

\subsection{Corpus}
\label{sec:corpus}

\begin{itemize}
  \item \textbf{Backbone: OpenWebText \citep{Gokaslan2019OpenWebText}, superseding
    WikiText-103.} WikiText-103 (deduplicated) tokenizes to only
    $\approx$110M tokens total -- far short of the total-training-token target
    below, and, since \texttt{assemble\_corpus.py} reads the backbone once
    straight through rather than looping it, that number is a hard ceiling on
    how much of it could ever be packed into the on-disk corpus. OpenWebText
    has $\approx$9.04B tokens (9{,}035{,}582{,}198, per its published token
    count), comfortably enough for a single, non-repeating pass at the token
    budget below.
  \item \textbf{Streamed and consumed in bounded chunks, not downloaded in
    full.} OpenWebText is 38GB of raw text; the corpus this project actually
    needs (\S\ref{sec:scale} below) is a fraction of that. The backbone loader
    (\texttt{preprocess/backbone\_openwebtext.py}) pulls documents from the
    \texttt{Skylion007/openwebtext} HF dataset via
    \texttt{datasets.load\_dataset(..., streaming=True)}, buffers raw text
    in memory only until a chunk boundary (default 2GB of accumulated text,
    configurable) is reached, hands that chunk to the assembler for
    tokenizing and packing (\S\ref{sec:assembly}), then discards the buffer
    and pulls the next chunk -- nothing beyond one chunk's raw text is ever
    held at once, and no persistent local copy of the dataset is written to
    disk. This is a storage-constraint requirement, not a performance
    optimization: the download must happen at run time, inside the
    assembly/training job itself, never as a separate pre-staged local
    dataset.
  \item Injected data: synthetic biography documents, one per entity
    (\S\ref{sec:entity}), inserted at randomly scheduled positions in the
    backbone token stream (\S\ref{sec:scheduler}).
  \item Independent variable: exposure count only. Each entity has exactly one
    contested relation with two candidate values $A$ and $B$; the swept
    quantity is the split $(n_A, n_B)$ of a fixed total exposure budget $T$
    between the two sides (\S\ref{sec:scheduler}, \S\ref{sec:scale}).
    \textbf{Recency is not tracked or manipulated} in this design: in a
    from-scratch training run, an occurrence's position late in the stream is
    confounded with the cosine LR schedule's decay toward that point, so
    "recency" as previously defined would have been entangling exposure
    timing with optimization dynamics rather than isolating it. This is a
    deliberate scope decision, not an oversight, and supersedes every
    recency-bin mechanism in the prior design.
  \item \textbf{Token budget: fixed at $2.5\times10^9$ total training tokens},
    the Chinchilla-optimal point for a 124M-parameter model
    \citep{hoffmann2022chinchilla} at $\approx$20 tokens/parameter --
    superseding the earlier $\mathcal{O}(10^8\text{--}10^9)$ placeholder range
    now that a concrete backbone (OpenWebText) can actually support it. Per
    \S\ref{sec:corpus-vs-training-tokens}'s distinction, this is the total
    training-token figure (batch $\times$ context $\times$ steps,
    \S\ref{sec:optimization}), not necessarily the on-disk corpus size: since
    OpenWebText has enough unique tokens to cover $2.5\times10^9$ in a single
    pass with no repetition, \texttt{total\_tokens} (on-disk, S1.7) is set
    equal to the training-token target rather than smaller, so the run needs
    zero epochs of repetition over its own backbone -- the cleanest point on
    \citet{muennighoff2023scaling}'s curve.
\end{itemize}

\subsection{Entity and Document Construction}
\label{sec:entity}

\begin{itemize}
  \item \textbf{Entity schema}: each synthetic entity is a tuple
    $(\text{name}, \text{contested}, \{\text{background}_j\})$. \emph{Name}
    is drawn from a globally unique entity-name pool, disjoint from every
    other human-name-shaped value pool. \emph{Contested} is one relation
    type (relation-type assignment, below) assigned two candidate values
    $\text{val}_A, \text{val}_B$ from that relation's value pool and a split
    $(n_A, n_B)$, $n_A + n_B = T$ (\S\ref{sec:scheduler}). $\{\text{background}_j\}$
    is a fixed-size set of $K=4$ further relation-type/value pairs, each
    assigned a single value that never varies -- these give the biography a
    natural bioS-parallel length (1 contested $+$ 4 background $=$ 5
    attributes per entity) and reinforce the entity's identity across
    occurrences: background content is restated every time \emph{any}
    occurrence of the entity's biography renders (\S\ref{sec:scheduler}), so
    its exposure count is exactly $T$, not independently controlled.
  \item \textbf{Relation-type assignment, balanced by construction, not a
    designed grid}: entities are compiled directly from the name/value pools
    rather than populated into an explicit per-cell configuration table, but
    usage is still exactly balanced, not independently random per entity:
    \begin{itemize}
      \item \emph{Contested type}: one shuffled list of (relation type,
        split level) pairs is built, each of the 14 $\times$ (number of
        split levels) combinations repeated the same replicate count
        (\S\ref{sec:scale}); one pair is assigned per entity by walking the
        shuffled list. This guarantees every relation type is used as the
        contested type the same number of times overall, and the same
        number of times within every split level, so relation-type identity
        cannot confound the frequency-gap comparison.
      \item \emph{Background types}: a fixed cyclic offset from the
        contested type -- given a fixed ordering of the 14 relation types,
        an entity whose contested type sits at position $c$ takes background
        types at positions $c{+}1, c{+}2, c{+}3, c{+}4$ (mod 14). Because the
        contested-type assignment above is exactly balanced, this offset
        rule makes every relation type's background-usage count exactly
        balanced too, with no separate randomization needed. (This does make
        ``contested $=X$'' and ``background includes $X{+}1..X{+}4$''
        co-occur deterministically rather than just typically -- acceptable
        since background identity is reinforcement, not itself an analyzed
        variable.)
      \item \emph{Values}: for a given relation type, values are drawn by
        walking a shuffled copy of that relation's value pool in order,
        reshuffling and starting a new lap once the pool is exhausted, so
        usage counts across the pool differ by at most one rather than
        drifting under i.i.d.\ resampling. The contested pair's
        $\text{val}_A, \text{val}_B$ are drawn this way and checked for
        surface-form divergence (\S\ref{sec:divergence}); a failing pair
        advances to the next pool position rather than resampling blind.
    \end{itemize}
  \item \textbf{One contested relation per entity, not several}: an earlier
    version of this design considered giving a single entity multiple
    simultaneously-contested relations, reasoning that it would amortize the
    name-introduction cost further. This was rejected: whenever the
    entity's biography renders to advance one contested relation's exposure
    count, every other attribute in that same instance -- including any other
    contested relation -- is restated too, coupling the two conflicts'
    exposure patterns through shared document instances. That reintroduces,
    across two \emph{contested} relations, exactly the cross-relation
    entanglement that keeping relations independent was meant to avoid, and
    is worse than the analogous issue for background relations because both
    dependent variables (which side wins for each relation) would become
    correlated by construction. One contested relation per entity keeps each
    conflict pair's exposure pattern clean.
  \item \textbf{Relation-type inventory} (14 types, spanning affiliation,
    fixed life-event, and ongoing-state semantics so findings are not an
    artifact of one relation's surface statistics): alma mater; employer;
    birthplace; current residence (distinct value
    space from birthplace); award or honor received; authored work or
    invention (title of book/patent/device); professional or hobbyist
    affiliation (society, guild, club); field of expertise or specialization;
    mentor or doctoral advisor (named individual, drawn from a name pool
    disjoint from all entity names); primary funding source or grant-making
    body; professional license or certification held; publication venue
    (journal or imprint); civic or volunteer organization; working or
    professional language. Each relation type has its
    own value pool, checked against named entities in WikiText-103 and against every other
    value pool.
  \item \textbf{Value-pool reuse policy}: a value may be reused across
    different entities if it is not itself shaped like a human name
    (institution, place, award, license, venue, and similar values are drawn
    from finite pools and may repeat across entities, matching real-corpus
    statistics). Any value that is itself a person's name -- the entity's own
    name, or the mentor/advisor relation's value -- is drawn from a globally
    unique pool and never reused across entities. The entity-name pool and
    every other human-name-shaped value pool are each constructed as a
    separate bank (e.g., via a dedicated auxiliary-LLM call per pool) and
    checked for mutual disjointness.
  \item \textbf{Document rendering}: LLM-authored, not mechanically assembled
    from templates. Operationalized as bioS(N) in spirit
    \citep{allenzhu2023physics} -- a full biography per (entity, side) rather
    than a global "first mention" elsewhere in the corpus -- but the text
    itself is written by an auxiliary LLM (\S\ref{sec:generation}) given the
    entity's name and its 5 facts (contested relation at one specific side,
    plus all $K$ background relations) handed over \emph{flat}: the prompt
    gives no indication that one fact is contested, so the model has no
    opportunity to hedge, editorialize, or otherwise write differently about
    it than about an ordinary background fact. One call per (entity, side)
    produces $V$ independent variants of a short, coherent paragraph stating
    all 5 facts (\S\ref{sec:generation} for the exact prompt and $V$'s
    default). This replaces the prior first\_mention/restatement mechanical
    split entirely -- there is no separate template bank per relation type,
    and no notion of one attribute "opening" the instance while others are
    "restated"; the model composes the whole paragraph, choosing its own
    structure and word order freely each variant.
  \item \textbf{Value fidelity is the one hard constraint on that freedom}:
    every fact's value must appear verbatim in the generated text (enforced
    by prompt instruction, \S\ref{sec:generation}, and checked post-hoc --
    a variant where the exact string cannot be located is rejected and
    regenerated), because per-fact recall (\S\ref{sec:eval}) is scored by
    locating that exact string in the rendered text and reading the model's
    probability at that point. A paraphrased-away value (e.g. "a coastal
    town" instead of "Aldergrove") leaves nothing to score. Everything else
    about the paragraph -- structure, sentence count, which fact leads -- is
    left to the model's discretion. The post-hoc check is
    case-\emph{insensitive}: a real common-noun value (e.g.\ a
    \texttt{field\_expertise} term like "Paleoclimatology") is reliably
    case-folded to lowercase mid-sentence by a fluent writer, unlike the
    invented proper-noun values (places, institutions, people), which keep
    their capitalization. Exact case was never the actual requirement --
    only that the value's position in the text is findable for scoring --
    so case is ignored rather than added as a second name-shaping
    constraint on top of \S\ref{sec:entity}'s existing one. At population
    scale, the pre-fix exact-case check caused a $\sim$28\% total-failure
    rate (0 usable variants after all retries), concentrated in exactly the
    entities carrying a \texttt{field\_expertise} fact -- diagnosed and
    fixed via the resumable batch run, not caught by the single-example
    pilot (\S\ref{sec:generation}, which happened not to include that
    relation type).
  \item \textbf{Values must be name-shaped, not full descriptive phrases}:
    piloting the vignette prompt surfaced that a fluent LLM writer reliably
    keeps short, name-shaped values verbatim (a place, an award, a society
    name) but naturally rewords a full descriptive clause into different
    prose -- e.g. "a senior analyst at Ashgrove Dynamics" became "works at
    Ashgrove Dynamics as a senior analyst," silently breaking value fidelity
    above. \texttt{alma\_mater\_values}, \texttt{employer\_role\_values}, and
    \texttt{civic\_role\_values} were shortened to bare institution/
    organization names for this reason (\S\ref{sec:generation});
    \texttt{civic\_role\_values} was the last holdout (originally deferred
    rather than fixed alongside the other two) and its full-phrase shape
    (e.g.\ "board member of X") was confirmed, not just suspected, as the
    sole cause of a 152/3{,}360-pair (4.5\%) vignette generation failure
    cluster at population scale before being fixed. Any future value pool
    must be name-shaped for the same reason, not merely fictional.
  \item \textbf{Variant choice is round-robin, not resampled}: at each
    occurrence event (\S\ref{sec:scheduler}), one of that side's
    pre-generated variants is used, drawn without repetition until all have
    been used once, then reshuffled -- the same discipline as the value
    assignment above, applied here to variant choice rather than fact
    content, so an entity's occurrences do not reuse the same paragraph
    back-to-back purely by chance. State is kept per entity, persisting
    across that entity's occurrence events, not reset per instance.
  \item \label{sec:variants-per-side} \textbf{Total variants per entity fixed
    at 8, allocated unevenly across sides via \texttt{variants\_per\_side}
    (\texttt{preprocess/scheduler.py})} -- superseding a flat $V$ split
    evenly (or near-evenly) between side $A$ and side $B$. Given the
    entity's own $(n_A, n_B)$ (\S\ref{sec:scale}), $v_B = \max(1,
    \mathrm{round}(8 \cdot n_B / (n_A{+}n_B)))$ and $v_A = 8 - v_B$: the
    side with more scheduled occurrences gets proportionally more fresh
    material before round-robin repeats kick in, rather than every entity
    receiving the same 4/4 (or similar) split regardless of how lopsided its
    own exposure gap is. At the balanced split $(40,40)$ this still gives
    $(4,4)$; at every split from $(68,12)$ onward the low side floors to a
    single variant, which is both the proportionally-implied allocation and
    the practical minimum (a side that occurs at all needs at least one
    usable paragraph). Combined with $T=80$ (\S\ref{sec:scale}), most
    occurrences necessarily reuse a small number of paragraphs many times
    over -- an explicit, accepted design choice (\S\ref{sec:assembly}), not
    an oversight.
  \item \textbf{Resolved: recall scoring is decoupled from training text}
    (was flagged here as an open question; see \S\ref{sec:assembly}).
    \S\ref{sec:divergence}'s single-token surface-form divergence guarantee
    was construction-guaranteed under shared templates (side $A$ and side
    $B$ were identical text except at the value); independently-authored
    free-text vignette variants have no such guarantee. Per-fact recall
    (\S\ref{sec:eval}) is scored on a separate, fixed template-based probe,
    not on training vignettes -- \S\ref{sec:assembly} has the full
    resolution.
\end{itemize}

\subsection{Pool and Vignette Generation}
\label{sec:generation}

Every name/value pool referenced in \S\ref{sec:entity}, and every entity's biography
text itself (\S\ref{sec:entity}, document rendering), is produced by an auxiliary LLM
(gpt-4o-mini via the OpenAI API, switched from GPT-4.1 on cost grounds -- $\sim$13x
cheaper on both input and output at standard rates, and neither pool/template phrasing
nor vignette-writing at this scale needs GPT-4.1-level capability) rather than
hand-authored, so pool/vignette content does not itself encode the experimenters'
assumptions about frequency. Pool generation uses
one meta-prompt per pool (15: the entity-name and mentor-name pools, and thirteen
relation-type value pools; Table~\ref{tab:pools}), substituting only a category
description and a target count into an otherwise fixed template:

\begin{quote}\small\ttfamily
Generate \{n\} distinct entries for this category: \{description\}\newline
Rules: each entry is \{name-shaped / short-phrase\}; \{fictional-required /
real-terms-allowed\}; no trailing punctuation/numbering/commentary; prefer a
distinctive first word per entry. [Do not repeat: \{already-used entries\}.]\newline
Output ONLY a JSON array of exactly \{n\} strings.
\end{quote}

Vignette generation (superseding the prior per-relation-type
first\_mention/restatement template meta-prompt, S1 revision (g)) uses one call per
(entity, side): the entity's name and its 5 facts (contested relation resolved to that
side, plus all $K$ background relations) are substituted flat into a fixed template,
with no indication that any one fact is contested:

\begin{quote}\small\ttfamily
Write a short biography of a fictional person for a synthetic training corpus.\newline
Person: \{name\}\newline
Facts (state every one of these somewhere in the biography):\newline
- \{relation label\}: \{value\} [$\times$5, one line per fact]\newline
Write \{n\_variants\} independent variants of this biography.\newline
Rules: each variant is a short, coherent paragraph (3--6 sentences), third person,
plain declarative prose; mentions \{name\}'s full name at least once and states every
fact above; each fact's value must appear verbatim, exactly as given (automatic
scoring later depends on finding the exact string); presents every fact flatly, as
simply true, with no hedging or editorializing; the \{n\_variants\} variants are
genuinely different from each other (opening fact, sentence order, phrasing), not
synonym swaps of one paragraph; invents no additional facts; fictional/generic
otherwise.\newline
Output ONLY a JSON array of exactly \{n\_variants\} strings, one full paragraph per
variant.
\end{quote}

\textbf{Generation procedure (pools)}: requests are issued in batches rather than one
call per pool, each batch sized at $\lceil 1.2 \times \text{remaining}\rceil$ (a 20\%
leeway over the strict remaining count, to absorb rejected duplicate/invalid entries
without a guaranteed extra round-trip) and capped at 150 entries per call, bounding
single-completion length; failed calls retry up to 5 times with exponential backoff.
A post-generation cleaning pass removes exact duplicates (case/whitespace-normalized),
malformed entries (wrong word count, trailing punctuation, LLM refusal artifacts), and,
for name pools, entries not shaped like "Firstname Lastname," before a pool is
accepted; the cross-pool and WikiText-103 disjointness checks of \S\ref{sec:entity} run
afterward, on the cleaned pools.

\textbf{Generation procedure (vignettes)}: one call per (entity, side), requesting
that side's target variant count (\texttt{v\_a} or \texttt{v\_b} from
\S\ref{sec:entity}'s \texttt{variants\_per\_side} -- 8 total per entity, split
unevenly across sides in proportion to $(n_A, n_B)$, not a flat $V$ for both sides
as in the superseded design); sampled at a moderately elevated
temperature (higher than the near-deterministic pool/template calls) so the
$V$ variants are genuinely structurally distinct rather than near-duplicates. Each
returned variant is checked post-hoc for the value-fidelity requirement above (exact
substring match for every one of the 5 values); a variant failing this check is
dropped and the call retried up to a bounded number of attempts before surfacing an
error rather than silently accepting a variant nothing can be scored against.

\textbf{Concurrency}: at population scale ($\sim$1{,}680 entities $\times$ 2 sides
$=$ 3{,}360 calls) requests are issued concurrently (async, bounded by a semaphore)
rather than one at a time, with a conservative default concurrency cap (10
in-flight requests) since actual account-level rate limits are not known in advance;
per-request exponential backoff handles transient/rate-limit errors without the whole
job failing on one throttled call. The run is resumable per entity -- an
(entity\_id, side) pair already written to disk is skipped -- so a job that
stops partway (rate-limit exhaustion, crash, or a deliberate stop) restarts without
redoing completed work, per CLAUDE.md \S5's restart-safety requirement for
long-running jobs. \textbf{As of this revision the run is incomplete} -- see the
STATUS note at the top of this file for the exact counts; resuming is just
re-invoking the same command.

\subsection{Surface-Form Divergence}
\label{sec:divergence}

Per-fact recall (\S\ref{sec:eval}) is scored at a single token to avoid a
length confound: comparing joint log-probability across candidate values of
differing token count penalizes the longer candidate independent of any
frequency effect (fewer multiplied sub-unity terms yield a higher joint
log-probability). Scoring at the first token where two candidate values
diverge avoids this, provided two requirements are enforced at generation
time rather than assumed:

\begin{itemize}
  \item \textbf{Construction}: templates are authored so no shared filler
    token sits between the template stem and the value -- the token stream
    diverges at the value itself, not several tokens into it.
  \item \textbf{Verification}: checked per rendered context, not per bare
    string, since BPE assigns a different token to the same word depending on
    leading-space / sentence-initial position (e.g., mid-sentence
    ``\texttt{ Meridian}'' vs.\ sentence-initial ``\texttt{Meridian}'' in a
    subject-first paraphrase).
\end{itemize}

An earlier version of this check verified every value against every other
value in the same relation-type pool -- an $O(n^2)$ requirement that,
empirically, rejected an unworkable fraction of a pool once $n$ grew into the
hundreds (a 150-entry pool lost 15--70\% per round; the 1200-entry
\texttt{mentor\_names} pool lost 58\%), stronger than the science needs. Under
the single-contested-relation-per-entity design (\S\ref{sec:entity}), this is
even more clearly true than before: exactly one pair of values (val$_A$,
val$_B$) is ever directly compared for a given entity, so verification is
three checks applied on demand, not a pool-wide pre-pass:
\begin{itemize}
  \item[(a)] \textbf{On-demand pairwise divergence}, applied to the two
    values assigned as an entity's contested pair, at the time that
    assignment is made -- not pre-verified against the rest of the pool.
  \item[(b)] \textbf{Exact-duplicate ban}, unconditional and independent of
    (a)/(c) (enforced at generation time, \S\ref{sec:generation}).
  \item[(c)] \textbf{No overly common first token}: a value is rejected if
    its distinguishing first token (post-stem) decodes to fewer than 3
    characters (e.g.\ ``\texttt{ A}'', ``\texttt{ Th}''). Rare/invented words
    disproportionately fall back to such short BPE subwords, which are shared
    by a large fraction of the vocabulary and therefore likely to collide
    with whatever a value is eventually paired against -- an $O(n)$
    per-value proxy rather than a corpus frequency lookup.
\end{itemize}
Templates are still authored per (a)'s construction rule above; only the
pool-wide pre-verification requirement was relaxed.

\textbf{Status under LLM-authored vignettes (S1 revision (g), resolved in
\S\ref{sec:assembly})}: this section's checks were designed around, and are still
run against, a shared \emph{template} (the mechanism above still gates which
val$_A$/val$_B$ pairs are even offered to the vignette generator,
\S\ref{sec:generation}). They no longer guarantee that the \emph{training text}
itself diverges at one shared token position, since side $A$'s and side $B$'s
paragraphs are independently authored and may differ in wording well before the
value -- so this construction is repurposed to hold only for the separate
eval-time probing format (\S\ref{sec:assembly}), not for training text.

\subsection{Exposure Scheduler}
\label{sec:scheduler}

Supersedes the prior recency-bin Insertion Scheduler entirely. Fact placement
is generated by a scheduler decoupled from document content, mapping each
entity's contested-relation split $(n_A, n_B)$ to token positions in the full
backbone-plus-injection stream:

\begin{itemize}
  \item \textbf{Two threads, one document}: side $A$ and side $B$ of an
    entity's contested relation are two \emph{threads} that share the same
    underlying biography-rendering machinery (\S\ref{sec:entity}), rather
    than two independent documents. Thread $A$ contributes $n_A$ occurrence
    events, thread $B$ contributes $n_B$, $n_A + n_B = T$.
  \item \textbf{Placement}: all $T$ occurrence events for an entity are placed
    i.i.d.\ uniformly at random across the full planned training-token
    stream, then labeled $A$ or $B$ according to which thread they belong to.
    No recency bins, no last-occurrence jitter-within-a-bin: placement
    timing is not a studied variable (\S\ref{sec:corpus}), so a single
    well-understood uniform randomization is used rather than any
    position-dependent mechanism.
  \item \textbf{Rendering coupling}: at each of the $T$ occurrence events
    (processed in chronological order), the entity's full biography instance
    is rendered (\S\ref{sec:entity}) with the contested relation resolved to
    whichever side that event belongs to. Every other attribute (all
    $K$ background relations) is rendered fresh at that same event
    regardless of which side fired -- this is what gives background
    content its exposure count of exactly $T$, with no separate scheduling
    required.
  \item \textbf{Logging}: the exact position and side-label of every
    occurrence event is recorded per entity, sufficient to reconstruct
    $(n_A, n_B)$ and total exposure count at analysis time. No recency
    measures are computed (superseded).
\end{itemize}

\subsection{Corpus Scale: Conflict-Pair Phase}
\label{sec:scale}

\begin{itemize}
  \item \textbf{Split levels: $T=80$, 6 levels}, superseding the earlier
    $T=32$ placeholder -- \textbf{a scaled-up working estimate adopted ahead
    of the exposure-budget pilot (\S\ref{sec:seeds}), not itself a
    pilot-validated value.} Rationale for raising it: a capacity argument
    (\S\ref{sec:seeds}) shows this population is nowhere near
    capacity-constrained, so Allen-Zhu's 1000-exposure regime is the wrong
    anchor, but $T=32$ likely undershoots the point where a contested pair's
    confidence gap is reliably learned rather than noisy; the pilot is still
    what fixes this properly, and should sweep around this new estimate
    rather than assume it. Split levels (all summing to $T=80$, same
    balanced-to-extreme spanning shape as the $T=32$ table it replaces):
    $(40,40)$, $(50,30)$, $(60,20)$, $(68,12)$, $(73,7)$, $(77,3)$.
  \item \textbf{Replication}: 20 replicate entities per (relation type, split
    level) combination (14 types $\times$ 6 levels $\times$ 20 $=$ 1{,}680
    entities total) -- unchanged in count from the $T=32$ population; the
    $T=32\to T=80$ migration remapped each entity's existing $(n_A, n_B)$ to
    its new value at the same split-level index (preserving every entity's
    name, contested relation type, values, and background facts exactly, so
    the population did not need to be rebuilt from scratch). Relation type
    and split level are both exactly balanced by the construction in
    \S\ref{sec:entity} -- every relation type is contested the same number
    of times within every split level, so relation-type identity cannot
    confound the frequency-gap comparison. Background-relation-type usage is
    also exactly balanced, via the fixed cyclic-offset rule of
    \S\ref{sec:entity}.
\end{itemize}

\subsection{Corpus Assembly}
\label{sec:assembly}

Interleaves the OpenWebText backbone (\S\ref{sec:corpus}) with entities' vignette
occurrences (\S\ref{sec:scheduler}) into the packed token shards
\texttt{training/data.py:PackedTokenDataset} already expects (\texttt{meta.json} +
\texttt{*.bin}), plus a separate occurrence log.

\begin{itemize}
  \item \textbf{Single streaming pass, not two passes over a local file} --
    superseding the WikiText-103-era design entirely. That design required a
    fully-materialized local backbone file to read twice (once to measure
    line lengths, once to write tokens); OpenWebText is instead pulled
    through \texttt{preprocess/backbone\_openwebtext.py}'s chunked HF stream
    (\S\ref{sec:corpus}), so there is no stable local file to re-read a
    second time. This is not just a workaround but a simplification: every
    occurrence event's position (\S\ref{sec:scheduler}) is already a
    real-valued coordinate into $[0, \texttt{total\_tokens})$ decided purely
    from the target token budget, with no dependency on backbone content --
    so all events can be precomputed and sorted up front, then fired online
    while streaming. \textbf{Two counters are kept, deliberately not one}: a
    backbone-only cumulative count (compared against event positions, since
    those coordinates are defined against the backbone stream alone,
    \S\ref{sec:corpus-vs-training-tokens}) and the actual full packed-stream
    offset including already-injected tokens (used only for the occurrence
    log). Comparing events against the full-stream offset instead would let
    an injected vignette's own tokens push that counter past \emph{later}
    events' positions, firing them in a cascade well before the backbone
    content that was supposed to precede them had actually been written --
    caught in a mocked dry run (no real OpenWebText access needed to catch
    it) before this shipped.
  \item \textbf{Document boundaries, and insertion granularity within them}:
    each OpenWebText row is one self-contained article (the natural document
    unit here, unlike WikiText-103's per-line convention) and each injected
    vignette occurrence is its own self-contained document, both delimited by
    the GPT-2 EOS token. Injections are never spliced mid-sentence, but they
    are also not restricted to firing only between whole articles: OpenWebText
    documents run up to $\sim$100k characters, so checking only at document
    boundaries would let a scheduled position sit unfired for an entire long
    article before firing, well past where it was meant to land. Instead each
    document is tokenized once, at full length
    (\texttt{backbone\_openwebtext.py:tokenize\_with\_unit\_boundaries} --
    tokenizing paragraph fragments separately and concatenating them was tried
    and rejected: it silently drops the whitespace between paragraphs and
    breaks any BPE merge that should span the boundary, verified concretely
    during review), which also returns paragraph- (or sentence-, fallback)
    boundary token indices; the event-firing check runs after every such
    boundary, not just after every whole document. Two consequences of firing
    mid-document rather than only between documents: (1) EOS is appended only
    after a document's final boundary, so an uninterrupted article stays one
    EOS-delimited unit on disk; (2) a vignette that fires at an
    \emph{internal} paragraph boundary gets an EOS forced immediately before
    it (in addition to the one already following it) -- without this, the
    vignette would run directly into the preceding backbone paragraph with no
    separator at all, exactly the concatenation this design is meant to avoid.
    The rare case of a vignette firing exactly on a whole-document boundary
    (which already ends in EOS) produces a harmless doubled EOS rather than
    needing special-cased detection.
  \item \textbf{\texttt{total\_tokens} is a backbone budget, not the final
    corpus length}: backbone lines are tokenized and accumulated until this
    budget is met; scheduler positions are coordinates into that
    backbone-length stream. Injected occurrences add their own tokens on
    top, so the realized corpus is slightly longer than
    \texttt{total\_tokens} by the total injected-token count. This avoids a
    circular dependency (exact final length depends on injected-text
    lengths, which vary per LLM-authored variant and aren't known until
    generation time).
  \item \label{sec:corpus-vs-training-tokens} \textbf{\texttt{total\_tokens}
    (on-disk corpus size) and total training
    tokens (\S\ref{sec:optimization}) are two independent quantities, not the
    same number under two names}: this section's \texttt{total\_tokens} fixes
    how many distinct tokens exist in the packed shards on disk. Total
    training tokens is \texttt{batch\_size} $\times$ \texttt{n\_positions}
    $\times$ \texttt{max\_steps} (\S\ref{sec:optimization}) -- the number of
    tokens the optimizer actually processes over the whole run, which
    routinely exceeds \texttt{total\_tokens} because
    \texttt{training/data.py:PackedTokenDataset} is sampled with replacement,
    not iterated as fixed epochs: \texttt{training/train.py:infinite\_loader}
    reshuffles and re-cycles the same fixed set of on-disk windows
    indefinitely once one pass is exhausted, rather than stopping when the
    corpus runs out. Concretely, a corpus with $N$ on-disk tokens trained for
    $M$ total training tokens revisits that corpus $\approx M/N$ times over
    the run -- this ratio is exactly the ``WikiText exposure'' open parameter
    of \S\ref{sec:seeds}, and must be kept small enough that repetition
    itself doesn't degrade quality (\citet{muennighoff2023scaling}: negligible
    loss up to $\sim$4 epochs, a $\sim$16-epoch half-life past which returns
    diminish sharply).
  \item \textbf{Chunked consumption, memory-bounded either way}: each
    $\sim$2GB text chunk from \texttt{backbone\_openwebtext.py} is tokenized
    and its documents written directly to the output shard as they're
    produced, then the chunk is discarded before the next one is pulled --
    never holding more than one chunk's raw text, or the full token stream,
    in memory at once. (The prior WikiText-103-era design's two-pass
    approach solved a different problem -- avoiding a second full read of a
    large local file -- that doesn't arise here, since OpenWebText is never
    fully local to begin with, \S\ref{sec:corpus}.)
  \item \textbf{Injected-token floor and its consequence for
    \texttt{total\_tokens}}: the number of occurrence events is fixed by
    population size and $T$ (1{,}680 entities $\times$ $T$, independent of
    \texttt{total\_tokens} -- it only controls how spread out they are, not
    how many there are), so there is a fixed injected-token floor that
    \texttt{total\_tokens} must be large relative to for the "rare
    conflicting fact in an otherwise natural corpus" framing to hold.
    At $T=80$ (\S\ref{sec:scale}) and $\sim$103 tokens/occurrence (matching
    the $T=32$-era measurement, gpt-4o-mini vignettes): $1{,}680 \times 80 =
    134{,}400$ occurrences, $\approx$13.8M injected tokens -- $\approx$0.55\%
    of the fixed $2.5\times10^9$ token budget (\S\ref{sec:corpus}), safely
    within the "rare perturbation" framing without needing to push
    \texttt{total\_tokens} any higher for this reason alone.
  \item \textbf{Variant selection}: round-robin per (entity, side), same
    discipline as \S\ref{sec:entity} (\texttt{PoolCycler}) -- no variant
    repeats until that side's bank (\texttt{v\_a} or \texttt{v\_b} variants,
    8 total per entity split unevenly per \S\ref{sec:entity}) is exhausted,
    then reshuffled. \textbf{Heavy repetition is expected and accepted, not a
    bug}: at the high-occurrence side of an extreme split (e.g. $(77,3)$'s
    77-occurrence side with 7 variants), each paragraph is reused
    $\approx$11 times over that entity's exposure history. This is the
    explicit "combination of fresh vignettes and repetition" the $T=80$/$V=8$
    design accepts rather than paying for enough fresh variants to avoid it
    entirely (\citet{allenzhu2024capacity}'s bioR uses 40 variants/entity as
    an upper anchor, but generating that many per side at population scale
    was judged not worth the added API cost here).
  \item \textbf{Output}: the packed stream (\texttt{data/processed/<name>/}) plus
    \texttt{results/occurrence\_log.json} (small, tracked) recording every
    occurrence event's entity, side, and final absolute token position in the
    packed stream -- sufficient to reconstruct $(n_A, n_B)$ and exposure
    timing at analysis time (\S\ref{sec:scheduler}), and for the training
    loop to derive dense-checkpoint step windows (\S\ref{sec:checkpointing})
    from token position and batch/block size (a training-side computation,
    not performed here).
  \item \textbf{Eval decoupling (resolves the open question in
    \S\ref{sec:entity}, \S\ref{sec:divergence}, \S\ref{sec:eval})}:
    per-fact recall is \emph{not} scored against training vignettes. Side
    $A$ and side $B$ training text are independently LLM-authored and share
    no common stem, so there is no single shared token position to compare
    at. Instead, the original per-relation-type template banks
    (\S\ref{sec:generation}'s first\_mention/restatement templates --
    superseded for training-text generation, but still generated and kept
    on disk) are repurposed as a fixed, controlled, eval-only probing
    format: side $A$ and side $B$ are rendered through the identical
    template, diverging at exactly one token as originally designed
    (\S\ref{sec:divergence}). Training text stays free-form and diverse;
    evaluation runs on a separate, minimal, controlled artifact the model
    never saw during training.
\end{itemize}

\subsection{Optimization}
\label{sec:optimization}

\begin{itemize}
  \item Optimizer: AdamW (fused CUDA kernel when running on GPU), cosine LR
    schedule with warmup, weight decay 0.01.
  \item Precision: bf16 mixed precision, gradient clipping.
  \item \textbf{Batch size / step count: 128 / 38{,}146} -- batch size passed
    through two GPU-driven revisions since the original 48 (see the
    2026-07-06 and 2026-07-09 REVISION notes at the top of this file for the
    full chain): first raised to 192 for A100 utilization, then confirmed by
    a real cluster run to CUDA-OOM at 192 (only $\approx$8GB free of 80GB
    against a 9.2GB backward-pass buffer) and dropped to 128, which leaves
    $\approx$26GB of headroom. Architecture (12-layer/12-head/768-dim) and
    weight decay/cosine schedule shape still match
    \citet{allenzhu2023physics} for comparability; batch size, and (per the
    2026-07-09 revision below) grad\_clip and AdamW eps, do not. Step count is
    re-derived, as before, from the fixed $2.5\times10^9$-token target
    (\S\ref{sec:corpus}): at 512-token context, $128 \times 512 \times
    38{,}146 \approx 2.5\times10^9$ tokens. Since \texttt{total\_tokens}
    (on-disk) is also fixed at $2.5\times10^9$
    (\S\ref{sec:corpus-vs-training-tokens}), this remains $\approx$one epoch
    over the backbone, not a multi-epoch repeat.
  \item \textbf{Learning rate, warmup, gradient clipping, and AdamW eps: lr
    $6\times10^{-4}$, warmup\_steps 2000, grad\_clip 0.5, eps $10^{-6}$} --
    revised 2026-07-09 (see that REVISION note above for the full incident
    and rationale) after independent, irrecoverable loss spikes hit the T=80
    and T=1280 sweep conditions partway through training. lr reverts to the
    standard GPT-2-small/GPT-3-125M literature value
    \citep{radford2019language, brown2020gpt3} rather than this project's own
    sqrt-scaled derivation (\citet{malladi2022sdes}) from the batch-size
    chain above, which had compounded onto an already-mismatched small
    reference batch across two rescalings. warmup\_steps is raised for extra
    margin ($\sim$2\% $\to$ $\sim$5\% of \texttt{max\_steps}); grad\_clip is
    tightened and eps set explicitly (not left at AdamW's implicit
    $10^{-8}$ default) per \citet{wortsman2024smallscale}'s finding that a
    too-small eps can let a layer's effective LR blow up once gradient RMS
    shrinks under high-LR instability. batch\_size stays at 128: an empirical
    CUDA probe (\texttt{training/probe\_max\_batch\_size.py}) found the true
    max at 512-token context is 140, not enough headroom over 128 to justify
    re-deriving lr/warmup/step-count for a batch-size change too.
  \item \textbf{GPU-memory fit is now validated}, superseding the
    corresponding open question in the prior revision of this section:
    batch=128 has run to tens of thousands of steps across the $T$-sweep
    without a memory failure. The loss spikes that motivated the
    2026-07-09 revision above were a training-stability failure, not a
    memory one, and are addressed by the checkpoint-rotation/spike-guard/
    rollback mechanism in \S\ref{sec:checkpointing}, not by batch size.
  \item Estimated wall-clock time on a single A100 80GB: using nanoGPT's
    published single-A100 benchmark ($\sim$178,000 tokens/sec at 49\% MFU
    training this same 124M-parameter architecture, itself measured with a
    fused optimizer and \texttt{torch.compile} -- both now also used here,
    \texttt{training/train.py}) as the closest available reference point,
    $2.5\times10^9$ tokens $/ 178{,}000 \approx 3.9$ hours -- comfortably
    inside a single 12-hour cluster job (CLAUDE.md \S2), though this figure
    has not been validated against this project's own throughput (different
    context length, GPU variant, batch size) and should be treated as a
    planning estimate, not a commitment, until measured.
\end{itemize}

\subsection{Checkpointing}
\label{sec:checkpointing}

% REVISION (2026-07-07): single-slot overwrite policy adopted, superseding the
% dual-cadence sparse-rotation + dense-stratified-full-weight-retention design
% below in its entirety (training/checkpoint.py, training/config.py). Explicit
% request: at most one pair of weights (model + optimizer state) stored on disk
% the whole time, so N simultaneously-running conditions (the T-sweep,
% \S\ref{sec:seeds}) cost a small, constant amount of checkpoint storage
% regardless of run length, rather than accumulating a stratified sample of
% dense full-weight snapshots per run as before. Every full save -- sparse-cadence
% or the unconditional final save -- now overwrites the same
% \texttt{checkpoints/latest/} directory via a crash-safe rename-swap (write to a
% fresh temp dir, then two atomic directory renames to swap it in for
% \texttt{latest/} and discard the old one) rather than an in-place file
% overwrite, so a crash mid-write can never corrupt the one surviving checkpoint.
% The final save supersedes whatever sparse checkpoint preceded it by the same
% mechanism. \textbf{Accepted cost, not an oversight}: dense checkpointing no
% longer captures full weights at all -- only its metrics-only markers remain
% (still a fine-grained loss/eval timeline near occurrence events). This removes
% the capability the paragraph below originally described (retrospective
% sparse-grid or dense-window mechanistic probing across a run's history, e.g.
% for the logit-lens/activation-patching/linear-probe work of
% \S\ref{sec:eval}) -- only the single most recent checkpoint is ever
% inspectable after a run finishes or is interrupted.
%
% REVISION (2026-07-09): single-overwriting-slot policy above superseded by
% $N=3$ rotating slots plus a stability layer (\texttt{training/checkpoint.py},
% \texttt{training/config.py}'s \texttt{StabilityConfig},
% \texttt{training/train.py}, \texttt{training/finalize\_checkpoint.py}) -- see
% the REVISION note at the top of this file for the full T=80/T=1280
% loss-spike incident that motivated this. The single-slot design's storage
% rationale (\S\ref{sec:seeds}'s $N$ simultaneous conditions cost a small,
% constant amount of checkpoint storage) still holds at $N=3$ slots (a small,
% constant multiple of before, not a growing history), while no longer
% risking the one surviving checkpoint being an already-diverged state that a
% rollback can't recover from.

Rotating slots, step-indexed, with an automatic stability layer (supersedes
both the wall-clock checkpoint rule in CLAUDE.md \S5 and the single-slot
design above):
\begin{itemize}
  \item \textbf{Sparse}: full checkpoint (weights, optimizer state, RNG state)
    every $S$ steps, $S$ set to bound compute loss on crash to
    $\sim$30 minutes at measured throughput -- written to a free
    \texttt{slot\_N} (crash-safe temp-write-then-rename) until
    \texttt{num\_checkpoint\_slots}$=3$ exist, then evicts the oldest-by-step
    slot, rather than overwriting a single \texttt{latest/} directory.
  \item \textbf{Dense}: every few hundred steps within a fixed window
    following each contested-relation occurrence event (\S\ref{sec:scheduler}),
    logs eval metrics only. No full weights are ever captured on this cadence
    (see 2026-07-07 revision note above).
  \item \textbf{Spike guard}: per training step, a 50-step rolling mean of
    loss is tracked; if a step's loss is non-finite or exceeds
    $4\times$ that mean (once at least 20 steps of history exist), the
    optimizer update is skipped for that step (gradients zeroed instead of
    applied) rather than letting a corrupted gradient update Adam's moment
    estimates, and the spiked loss is excluded from the rolling mean.
  \item \textbf{Rollback}: after each eval, a streak counter increments
    whenever val\_ppl exceeds $2\times$ the best val\_ppl seen so far; once
    that streak reaches 3 consecutive evals, model/optimizer/scheduler/RNG
    state is reloaded from the most recent rotating checkpoint, the learning
    rate is multiplied by a $0.7\times$ decay factor (floor: 0.1 of base lr,
    reapplied every step so the cosine schedule's own recomputation can't
    silently undo it), and the step counter is rewound to match. Rollback is
    capped at 5 occurrences; once exhausted, training stops early
    (rather than looping indefinitely) and the run summary records that it
    ended via exhausted-rollback, not by reaching \texttt{max\_steps}. This
    rollback/LR-penalty state is process-local: it does not persist across a
    separate preemption/resume of the training script, only within one
    continuous run.
  \item \textbf{Final}: an unconditional full save at the true last training
    step, through the same rotating-slot mechanism as Sparse above.
  \item \textbf{Finalize}: once training ends (by reaching
    \texttt{max\_steps} or by exhausted-rollback early stop),
    \texttt{training/finalize\_checkpoint.py} renames the single
    lowest-val\_ppl retained slot to \texttt{checkpoints/best/} and deletes
    every other slot -- so exactly one checkpoint remains on disk per run
    once training is done, matching the single-slot design's storage
    footprint at rest, and that is what any downstream eval
    (\S\ref{sec:eval}) reads by default.
\end{itemize}

\subsection{Evaluation}
\label{sec:eval}

Run at every checkpoint:
\begin{itemize}
  \item Held-out WikiText-103 perplexity (LM coherence check).
  \item Per-fact recall: log-probability / rank of the target value's first
    divergent token (\S\ref{sec:divergence}), computed per side ($A$/$B$) of
    the contested relation for every entity. Scored at a single token rather
    than the full continuation to avoid a token-count length confound, and
    via log-probability rather than free-form generation because base
    (non-instruction-tuned) checkpoints cannot be reliably prompted for
    parseable short-answer output.
\end{itemize}

% REVISION (2026-07-07): checkpointing moved to a single-slot overwrite policy
% (\S\ref{sec:checkpointing}) -- only the most recent checkpoint is ever on disk,
% so the sparse-cadence-grid mechanistic-probing plan below is no longer
% executable after the fact. Retained here as a statement of intent for any
% future run that explicitly re-enables multi-checkpoint retention, not as a
% description of what the current pipeline can do.
Mechanistic probes (logit lens, activation patching, linear probes for
which-fact-is-preferred), where run, would use the sparse-cadence grid at
windows flagged by the recall curves -- not currently possible against a
completed/in-progress run under \S\ref{sec:checkpointing}'s single-slot policy.

\textbf{Non-conflict (background) recall test} (2026-07-07, \texttt{eval/recall.py}):
a dedicated evaluation focus on the $4$ background (non-contested) facts per
entity, deliberately separate from the contested-term calibration question
above -- background facts receive exactly $T$ exposures per entity regardless
of contested split level (\S\ref{sec:entity}), so this gives one clean
recall-vs-$T$ curve per relation type, decoupled from split-level conditioning
entirely, and serves as a check that training produced any fact learning at
all, independent of the harder frequency-calibration question. Uses the same
fixed-template probing stem as contested recall (\S\ref{sec:divergence}), but
scored against the \emph{whole} $50{,}257$-token vocabulary at absolute
probability -- an explicit design choice against restricting to a
same-relation-type candidate pool, so the model must outscore every plausible
completion, not just other entries from that relation's value pool. Metrics,
per relation type and pooled overall across all $6{,}720$ background probes
($1{,}680$ entities $\times$ 4 facts): \textbf{top-1 accuracy} (correct token
is the model's argmax), \textbf{top-5 accuracy} (correct token ranks in the top
5), and \textbf{logit margin} (signed: the model's winning margin over the
runner-up when its top pick is correct, or the deficit behind its actual top
pick when wrong -- computed from the same log-softmax row already used for
rank, so this is numerically identical to a raw-logit margin). The pooled
figure is the one compared across the $T$-sweep (\S\ref{sec:seeds}); per-record
\texttt{relation\_key}/\texttt{entity\_id} tags are retained in the output JSON
so results remain sliceable by relation type after the fact. Legacy full-vocab
mean/median rank and mean log-probability are kept as secondary context, not
replaced.

\textbf{Resolved (S1 revision (g), \S\ref{sec:assembly})}: the per-fact recall metric
above presumes the target value's first divergent token is well-defined and shared
across side $A$ and side $B$'s renderings -- true by construction under the prior
shared-template design (\S\ref{sec:divergence}), not guaranteed once training text is
LLM-authored independently per side (\S\ref{sec:entity}). Recall is scored on a
separate, fixed-template probing prompt (the original per-relation-type templates,
kept for exactly this purpose) rather than on sampled training vignettes -- see
\S\ref{sec:assembly} for the full rationale.

\subsection{Calibration Target}
\label{sec:calibration}

Both sides of an intra-memory conflicting pair are model-internal claims with
no external ground truth, so calibration cannot be assessed against ``which
answer is correct.'' The target is instead defined relative to the exposure
statistics the corpus generator itself controls (frequency,
\S\ref{sec:corpus}): a rational aggregator of that evidence should produce a
confidence over $\{A, B\}$ satisfying three checkable, qualitative properties
rather than an exact posited functional form (e.g.\ an exact Bayesian
posterior over exposure counts):

\begin{itemize}
  \item \textbf{Monotonicity}: confidence in the higher-frequency side moves
    monotonically as the frequency gap $n_A - n_B$ widens; no reversals as
    the gap is dialed up.
  \item \textbf{Symmetry at balance}: at the balanced condition ($n_A = n_B$)
    confidence sits near 50/50 rather than locking arbitrarily onto one side.
  \item \textbf{No premature saturation}: confidence does not approach the
    softmax extremes long before the frequency gap does (overconfidence).
\end{itemize}

\textbf{Metrics}, derived from the same per-side log-probabilities already
scored in \S\ref{sec:eval} (no separate eval pipeline): softmax the two
log-probabilities into a confidence over $\{A, B\}$; plot against the
frequency gap over the split-level sweep of \S\ref{sec:scale}; compute expected
calibration error against the chosen normative curve, and a
monotonicity-violation count. By default this is computed from the
\emph{logit gap} -- the raw logit difference between the two sides at their
divergent token (\S\ref{sec:divergence}), equivalent up to a sigmoid
transform to the softmax confidence above -- and both the gap and the full
per-side logits are recorded, so alternative derived metrics remain
reconstructable without rerunning the model.

\textbf{Regime note}: this metric is scored at the base (non-instruction-tuned)
checkpoint stage per \S\ref{sec:eval}. That is not purely a practical
necessity (base checkpoints cannot be reliably prompted for parseable
short-answer output) -- it also keeps the token-level probability distribution
itself informative, which is the object this section's metrics operate on.

\subsection{Exposure-Count Calibration and Seeds}
\label{sec:seeds}

\textbf{Exposure-budget pilot}: rather than adopting an exposure-count regime
from prior work, $T$ (\S\ref{sec:scale}) is meant to be calibrated
empirically to this project's own setting. \citet{allenzhu2024capacity}'s
$\sim$1000-exposure regime is a poor prior here: their numbers are calibrated
to a capacity-stressed setting (millions of entities competing for a fixed
parameter budget), whereas this design uses a much smaller, non-capacity-
constrained entity population, so any single fact is learned far faster and
values anywhere near 1000 exposures would be expected to saturate both sides
of a contested pair well before the frequency gap does -- directly violating
the ``no premature saturation'' property of \S\ref{sec:calibration}. The
pilot itself has not been run yet; in its place, $T=80$ (\S\ref{sec:scale})
is a working estimate reasoned from the same capacity argument rather than
measured: this population's total factual content is
$1{,}680 \times 5 \times \log_2(150) \approx 61{,}000$ bits (5 facts/entity,
$\sim$150-entry value pools), against GPT-2-small's $\sim$2-bit/parameter
capacity ceiling of $124\text{M} \times 2 \approx 248$M bits
(\citet{allenzhu2024capacity}'s own measured saturation rate) -- a
$\sim$4{,}000$\times$ slack, i.e.\ nowhere near capacity-constrained, so
substantially fewer than 1000 exposures should suffice, but $T=32$ likely
undershot that point. $T=80$ is a reasoned midpoint, not a validated one:
once the pilot is actually run, it should sweep a range spanning this
estimate (e.g.\ $32/64/128/256/512$) and inspect the resulting
logit-gap/calibration curve; $T$ is fixed at a value where that curve is
monotonic but not yet saturated, per \S\ref{sec:calibration}. Only the
conditions selected by this pilot are then re-run at the full corpus scale
and seed count of \S\ref{sec:scale}.

\textbf{Seeds}: multiple random seeds and data-ordering variants per
condition, to separate a stable property of the setup from a
single-trajectory artifact, staged the same way as the exposure-budget pilot
above (single-seed pilot identifies conditions of interest; only those are
re-run at full seed count).

\textbf{Pilot scope, updated}: three of the original four parameters here
are now resolved working values rather than open unknowns, each reasoned
through in the sections cited, though none are pilot-validated:
\begin{itemize}
  \item \textbf{\texttt{total\_tokens}} (\S\ref{sec:corpus},
    \S\ref{sec:assembly}): fixed at $2.5\times10^9$, the Chinchilla-optimal
    point for a 124M-parameter model \citep{hoffmann2022chinchilla} -- no
    longer an open range, now that OpenWebText (\S\ref{sec:corpus}) can
    actually supply that many unique tokens in a single pass.
  \item \textbf{Backbone exposure}: with \texttt{total\_tokens} set equal to
    the total-training-token target (\S\ref{sec:corpus}), the backbone is
    seen $\approx$once (no repetition), sidestepping the over/under-sampling
    confound this bullet used to flag -- resolved by construction rather
    than by a separate ratio decision.
  \item \textbf{$V$} (variants per entity/side, \S\ref{sec:entity}): fixed
    at 8 total per entity, split unevenly across sides via
    \texttt{variants\_per\_side} (\S\ref{sec:entity}, \S\ref{sec:scale}) --
    up from the flat $V=5$/side default, though still well short of
    \citet{allenzhu2024capacity}'s bioR anchor (40/entity); heavy
    round-robin repetition on the high-occurrence side of extreme splits is
    accepted as a cost/API-budget tradeoff (\S\ref{sec:assembly}), not
    eliminated.
  \item \textbf{$T$} (\S\ref{sec:scale}) remains the one genuinely open
    parameter: $80$ is a reasoned working estimate (capacity-slack argument
    above), not a pilot-validated value. The pilot, once run, should sweep
    around this estimate and may revise $T$ (and, if $T$ moves far enough,
    revisit $V$'s allocation and the split levels of \S\ref{sec:scale}).
\end{itemize}

\section{Baseline Methods}

Baselines are grouped by mechanism. This project's setting is
\emph{intra-memory} conflict: both candidate answers are already parametric,
so there is no external context to contrast against a prior. Methods
originally designed for \emph{context-memory} conflict (context vs.\
parametric) are marked as such; applying them here requires substituting a
disambiguating cue (e.g., a time qualifier) for the missing external context.

\subsection{Logit/Decoding-Level Contrastive Methods}
\label{sec:decoding}

Output-distribution-only methods, no internal access required. All but DoLa
presuppose a with-context / without-context contrast, substituted here with a
disambiguating-cue pair.

\begin{description}
  \item[CAD] \citep{shi2023trusting} Contrasts with- and without-context
    output distributions with a fixed amplification coefficient.
  \item[COIECD] \citep{yuan2024discerning} Token-level entropy gate on CAD;
    contrastive adjustment applied only where an entropy-based conflict signal
    fires.
  \item[AdaCAD] \citep{wang2024adacad} Replaces CAD's fixed coefficient with a
    per-step, Jensen--Shannon-divergence-derived weight.
  \item[CoCoA] \citep{khandelwal2025cocoa} Adds confidence-aware terms
    (entropy gap, contextual peakedness) to AdaCAD's divergence-based weight.
  \item[DoLa] \citep{chuang2023dola} Contrasts early/middle-layer and
    final-layer logit distributions within a single forward pass; no context
    required. Directly applicable to the intra-memory setting without
    adaptation.
  \item[ARR] \citep{jiang2026contextaware} Routes each decoding step between
    trust-prior, trust-context, and agree regimes based on the conflict
    signal, rather than presupposing context trustworthiness as CAD/COIECD/
    AdaCAD/CoCoA do. Closest existing precedent for arbitrating between two
    parametric candidates without an external anchor.
\end{description}

\subsection{Activation Steering Methods}
\label{sec:steering}

Vector-based residual-stream interventions. Contrastive-pair construction does
not require external context, so this family ports to the intra-memory
setting more directly than \S\ref{sec:decoding}: pairs are built from two
disambiguated prompts, each eliciting one side of a conflicting-fact pair.

\begin{description}
  \item[CAA] \citep{rimsky2024steering} Steering vector from the
    mean-activation difference between contrastive prompt pairs, added
    (scaled) into the residual stream. Reference baseline for this family.
  \item[SpARE] \citep{zhao2024spare} Steers sparse-autoencoder features rather
    than dense residual directions; reports the conflict signal is linearly
    decodable from mid-layer activations onward, in the context-memory
    setting. Decodability under intra-memory conflict (no context-derived
    feature to isolate) is untested and should be treated as a hypothesis.
  \item[K-CAST] \citep{valentino2025mitigating} kNN-based conditional
    steering: per-instance direction from nearest-neighbor lookup rather than
    a single fixed vector. Originally applied to a reasoning/content-effects
    task.
  \item[ContextFocus] \citep{anand2026contextfocus} Context-faithfulness
    steering; reports composability with prompting-based methods. Requires
    the most adaptation of this group, being a context-faithfulness method by
    construction.
\end{description}

\subsection{Architecture/Attention-Level Intervention}
\label{sec:attention}

\begin{description}
  \item[PH3] \citep{jin2024cutting} Path-patching identification of attention
    heads causally responsible for propagating a memorized fact, followed by
    pruning/down-weighting at inference time. No context-vs-parametric
    contrast required; with DoLa, one of two methods surveyed here that is
    intra-memory-native by construction. Closest prior art to this project's
    CMAP intervention.
\end{description}

\subsection{Conflict-Detection Signals}
\label{sec:detection}

Detection only -- no output-changing mechanism.

\begin{description}
  \item[DynamicQA] \citep{marjanovic2024dynamicqa} Semantic entropy over
    paraphrase completions (adapted from Kuhn et al.) as an intra-memory
    conflict estimator; reports higher conflict for temporally/disputably
    dynamic facts than for static facts.
  \item[Self-detection] \citep{zhao2024knowing} Answer divergence across
    paraphrased questions (``memory strength'') as a signal for questions the
    model does not reliably know.\footnote{Yukun Zhao et al.; not to be
    confused with Yu Zhao, first author of SpARE (\S\ref{sec:steering}).}
\end{description}
Either signal is a candidate gate for the causal interventions in
\S\ref{sec:attention}, e.g.\ restricting PH3/CMAP-style pruning to
high-entropy tokens. The calibration target of \S\ref{sec:calibration}
additionally licenses a comparison unavailable to either source paper, which
lacks access to ground-truth exposure statistics: whether semantic entropy or
answer-divergence tracks the true frequency evidence gap as well as, or
better than, raw token-level confidence does.

\subsection{Calibration-Targeted Interventions}
\label{sec:calibration-interventions}

Methods that act on the calibration target of \S\ref{sec:calibration}
directly -- shaping how confident the model is -- rather than on which side
wins.

\begin{description}
  \item[Confidence penalty] \citep{pereyra2017regularizing} Training-time loss
    term penalizing low-entropy (over-peaked) output distributions, applied at
    the token positions where the two sides of a conflicting pair diverge. No
    architecture change, no context/cue required.
  \item[Temperature scaling] \citep{guo2017calibration} Post-hoc, single-scalar
    logit rescaling fit after training. Inference-time only, does not alter
    the model's factual content; the natural minimal baseline against which
    confidence-penalty training is compared.
  \item[Minimal in-context cue] Restating one side of the conflicting pair
    immediately before the query -- scaled down from the disambiguating-cue
    baselines of \S\ref{sec:decoding} to a form a base (non-instruction-tuned)
    model can plausibly use via induction-head-like copying rather than
    instruction comprehension. Tests whether a single in-context repetition
    shifts confidence consistently with, or in override of, the training-time
    exposure signal.
\end{description}

\subsection{Sampling/Ensemble-Based Mitigation}
\label{sec:sampling}

\begin{description}
  \item[Self-consistency] \citep{wang2022selfconsistency} Majority vote over
    multiple sampled completions; originally for chain-of-thought reasoning.
    Applied to factual recall: sample paraphrases of a probe question, vote
    or abstain on disagreement. Not evaluated in the literature as an
    intra-memory mitigation method specifically (only as a detection signal,
    \S\ref{sec:detection}); requires multiple forward passes per query, no
    internal access.
\end{description}
